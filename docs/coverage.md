# CI Gate — Coverage & Threat Model

What the `grizzly-gate` image actually prevents, given the configs currently in `config/`. This is the threat-coverage companion to the [design overview](../README.md) and the [operator runbook](https://github.com/Grizzly-Endeavors/grizzly-platform/blob/master/docs/runbooks/ci-gate.md). For an at-a-glance grid of the same coverage — threat class × language — see the [coverage matrix](coverage-matrix.md). Every claim here is traceable to a specific manifest, lint level, or scanner config in the tree — when the config changes, this doc must change with it.

The gate is the reviewer: a green gate is what lets an agent (or a human) ship without a second pair of eyes on the diff. So the question this doc answers is "what can no longer reach production unnoticed?" — broken into the structural guarantees that hold regardless of language, then the per-tool coverage, then an honest list of what the gate does *not* catch.

## Threat model in one paragraph

The gate sits between "an agent or contributor produced code + a built image" and "that image runs in the cluster." It assumes the author may be careless, may be an LLM taking every shortcut not explicitly denied, or may be actively hostile to the quality bar — both *weakening* the checks (relaxing the repo's own lint config, which config-forcing blocks) and *escaping* their scope (hiding code in a subdir, behind `.gitignore`, or in an un-adapted language, which the mandatory `gate-config.json` honest map blocks). It does **not** assume a compromised gate image or a compromised signing key — those are supply-chain trust roots handled by pinning + the secret store + Kyverno, not by the checks themselves. The output is a single cosign signature over the image **digest**; everything downstream trusts that signature and nothing else.

## Structural guarantees (independent of language)

These are the properties the harness and image enforce no matter what code is being gated. They are the load-bearing ones — a gap here defeats every per-tool check below.

- **Fail-closed on empty/missing config.** `load_tree` (`harness/src/config.rs`) bails if the config root yields zero manifests (or the required `detect.toml` is absent), and `main.rs` bails if `run_checks` produced zero results. A gate that ran nothing can never report PASS. This blocks the "delete the rules to pass the gate" and "point the gate at an empty dir" failure modes.
- **Honest-map verification — scope cannot be escaped.** Every gated repo must ship a root `gate-config.json` declaring its projects (missing/malformed ⇒ fail closed), and the harness (`detect.rs`) independently walks the *whole* tree and fails closed on any **undeclared** adapter-backed code (a `.rs`/`.py`/`.go`/TS/JS/`.svelte` file no declared project covers) or any **unsupported** language (code the gate has no adapter for — Ruby, Java, …). The walk scopes to the clean-checkout content (the git index — tracked files plus untracked-but-not-ignored files, exactly what a fresh `git clone` holds), does not follow symlinks, matches extensions case-insensitively, and shebang-classifies extensionless scripts — so a repo cannot hide code in a subdir, behind `.gitignore` (a *tracked* file is detected even when it matches an ignore pattern — `git add -f` does not help), under a renamed extension, or in an un-adapted language. On a non-repo source it falls back to a hostile raw-filesystem walk, which is strictly more inclusive. There are **no repo-controlled exclusions** (the master evasion key); only an Ops-owned `skip_dirs` list (deps/build/VCS) is skipped, and those paths are still secret/SAST-scanned. Adapters then run **per declared project** (in each project's dir), not just at the root. This is the guarantee that makes "green gate = every line was checked" true rather than "the code at the root was checked" — it closes what was previously the largest gap (see [ADR-029](decisions/029-gate-config-honest-map.md) and [ADR-036](decisions/036-honest-map-git-listing.md)). Adding a new language is therefore a deliberate two-part Ops change: an adapter under `languages/` *and* its detection rules — a repo cannot will a language into scope.
- **Config forcing — the repo cannot relax the rules.** Every adapter and scanner is invoked with the gate's own config injected via `{config}` (`--config`, `--config-path`, `--config-file`, `-c`, `CLIPPY_CONF_DIR`, `--no-config-lookup`). The scanned repo's own `.clippy.toml`, `ruff.toml`, `tsconfig.json`, `.yamllint`, `.gitleaks.toml`, `deny.toml`, etc. are ignored. This is the central anti-shortcut guarantee: an author cannot weaken the bar by editing config in their own repo, because that config is never read. Rust lint *levels* are forced on the command line specifically because `Cargo.toml [lints]` is applied earlier by rustc and would otherwise win.
- **Pinned tools; lint/SAST verdicts are reproducible, SCA verdicts are deliberately fresh.** Every tool version is pinned in the `Dockerfile` (no floating tags). The lint, SAST, and secret checks are deterministic: the Semgrep ruleset is vendored and the image Trivy DB is baked at build, so the same gate tag gives the same verdict for those. **The dependency-SCA scanners (`osv-scanner`, `trivy-fs`) intentionally do the opposite** — they fetch current advisory/license data at scan time, so a newly-disclosed CVE fails a previously-green build (this is the point of a supply-chain gate; see [ADR-030](decisions/030-cross-ecosystem-sca.md)). They fail closed if the data can't be fetched. So: same tag ⇒ same verdict for everything *except* SCA, where freshness is chosen over reproducibility. (Also: `SEMGREP_RULES_REF=develop` is a branch, not a SHA — flagged as a TODO in the Dockerfile.)
- **Signature binds to the digest, not a tag.** The gate signs `app@sha256:...`. A tag can be re-pointed at a different image after the fact; a digest cannot. This closes the "gate passes image A, deploy swaps in image B under the same tag" TOCTOU gap.
- **Signing happens only after all checks pass.** `--sign` is reached only past the `failed > 0` bail in `main.rs`. There is no code path that signs a failing build.
- **Self-contained signing.** `cosign sign --tlog-upload=false` — no dependency on (and no digest leakage to) the public Rekor log; verification is key-based against the public key embedded in the Kyverno policy.
- **Admission enforcement is separate from the gate.** Kyverno (platform-side, [`kubernetes/infrastructure/kyverno-policies/`](https://github.com/Grizzly-Endeavors/grizzly-platform/tree/master/kubernetes/infrastructure/kyverno-policies)) refuses unsigned images in namespaces labelled `grizzly.io/gated=true`. Even if someone pushes an unsigned or gate-skipped image straight to the registry, it cannot be admitted. The gate produces the proof; Kyverno is the wall.

## Per-tool coverage

### Secret scanning — gitleaks (source scope, always runs)

`gitleaks detect --redact --exit-code 1` over the source tree with the default ruleset (`useDefault = true`). Prevents committed credentials from reaching the repo/registry unnoticed: cloud keys (AWS/GCP/Azure), private keys, GitHub/GitLab tokens, Slack/Stripe/Twilio tokens, generic high-entropy assigned secrets, and `.env`-style leaks. `--redact` keeps the secret value out of CI logs so the scan itself doesn't leak it. Runs on every invocation regardless of language markers, so even a repo the gate has no adapter for still gets secret-scanned.

### SAST — semgrep (source scope, always runs)

`semgrep scan --error` against the vendored `semgrep-rules` set (offline). This is the broad source-level vulnerability net across languages. Concretely it catches the OWASP-style classes that rules exist for: injection (SQL/command/template), unsafe deserialization, weak/again-broken crypto (MD5/SHA1, ECB, hardcoded IVs), SSRF patterns, path traversal, insecure temp-file creation, use of `eval`/`exec` on untrusted input, disabled TLS verification, XXE, and framework-specific footguns. Because it runs unconditionally, it covers languages with no dedicated adapter too. (The same semgrep pass also enforces the **AI/LLM application-security** rules — significant enough for a family that builds AI apps that they get their own section immediately below.) (Note: the harness's own test suite carries a scoped `// nosemgrep: temp-dir` with a written reason — the sanctioned way to suppress a finding, not a blanket allow.)

### AI / LLM application security — semgrep `ai-best-practices` (source scope, always runs)

The gate is itself the control plane for AI-generated code, and the family ships AI applications — so the part of the semgrep ruleset that checks **how** an app uses an LLM is first-class here, not an afterthought. The vendored `ai/ai-best-practices/` set (~100 rules, all `category: security`, kept in full) runs on every invocation alongside the general SAST pass and blocks these classes across Python/JS/TS/Go/Java/Ruby and the generic (MCP/hooks) targets:

- **LLM output flowing into code execution** — `llm-output-to-exec` and `langchain-dangerous-exec`: a model's response reaching `exec`/`eval`/a shell. The highest-severity LLM bug class (model-controlled RCE).
- **Prompt injection via untrusted system prompts** — `*-user-input-in-system-prompt` for OpenAI, Anthropic, Gemini, Cohere, Mistral: user-controlled data concatenated into the system prompt, the root of most jailbreak/injection chains.
- **MCP server security** — `mcp-command-injection`, `mcp-ssrf`, `mcp-tool-poisoning`, `mcp-credential-in-response`, `mcp-unsanitized-return`, `mcp-hardcoded-config-secret`: the gate's own integration surface (it ships an MCP server) and every other MCP tool in the family held to injection/SSRF/credential-leak hygiene.
- **Claude Code hooks abuse** — `hooks-wget-pipe-bash`, `hooks-dns-exfiltration`, `hooks-sensitive-file-access`, `hooks-path-traversal`, `hooks-no-input-validation`, `hooks-unquoted-variable`, `hooks-unconditional-allow`, `hooks-relative-script-path`: the exact ways a hook (this repo ships several) becomes a code-exec or exfiltration vector.
- **Agent / settings safety** — `claude-settings-bypass-permissions`, `claude-settings-env-url-override`, `claude-settings-auto-enable-mcp`, `agent-unbounded-loop`, `ide-settings-executable-path`: config that silently widens an agent's authority or lets it spin unbounded.
- **Hidden-unicode instruction injection** — `ai-config-hidden-unicode`: invisible/bidi unicode smuggled into an AI config or prompt file (the "instructions you can't see" attack).
- **Hardcoded LLM credentials & missing safety controls** — per-provider `*-hardcoded-api-key` / `llm-api-key-in-source` (a provider-specific complement to gitleaks), plus robustness rules (`*-missing-moderation`, `*-missing-refusal-check`, `*-missing-safety-settings`, `*-missing-max-tokens`, `*-no-error-handling`).

What is deliberately **dropped at image-build** is the *other* half of the ruleset's `ai/` tree: 25 *shadow-AI inventory* rules (`ai/**/detect-*`, `category: maintainability`, severity `INFO`) that merely flag that a repo *uses* an AI SDK — including one generic-language rule that matches the literal strings `claude`/`anthropic` in **any** file. Under `--error` (every finding is a blocker) those would fail every legitimate AI-using repo and even this repo's own Claude-mentioning docs. Detecting *that* AI is present is not a security signal; the best-practice rules that check *how* it's used are. The prune is content-based (the `maintainability` category, unique to the inventory set within `ai/`) so an upstream rename can't reintroduce one — see the Dockerfile semgrep-vendoring step.

### Image CVE/SBOM — trivy (image scope, runs only with `--image`)

`trivy image --exit-code 1` failing on **HIGH** and **CRITICAL** vulnerabilities across both `os` and `library` package types. Prevents shipping a container whose base image or bundled dependencies carry known-exploitable CVEs. `ignore-unfixed: true` is a deliberate trade: unfixable vulns don't block deploys indefinitely (they aren't actionable at gate time), so the gate enforces "patch what's patchable," not "zero CVEs." This is the only check that inspects the built artifact rather than the source — it's what stops a clean-source repo from shipping a rotten base image.

### Rust dependency audit — cargo-deny (`Cargo.toml` marker)

`cargo deny check` with a fleet-wide config. **Vuln-only posture ([ADR-034](decisions/034-sca-vuln-only-license-loosening.md)):** blocks on RUSTSEC **vulnerabilities**, plus wildcard version requirements (`wildcards = "deny"` — they defeat reproducible resolution and are a supply-chain risk) and dependencies pulled from **unknown registries or git sources** (`unknown-registry`/`unknown-git = "deny"`, allow-list restricted to crates.io). **Does not** block on **unmaintained** crates (`unmaintained = "none"`) or **yanked** versions (`yanked = "warn"`) — those leave repos undeployable for non-security reasons a downstream often can't fix. The license allow-list is broadened to a wide permissive + weak-copyleft set so a dependency's license rarely blocks a deploy (strong copyleft GPL/AGPL is still absent — a deliberate, surfaced decision). A short, justified `ignore` list carves out individual **real** vulnerabilities with no upstream fix (currently `RUSTSEC-2023-0071`, the `rsa` crate's Marvin Attack timing sidechannel, pulled transitively via `sqlx-mysql` — no fixed release exists) — the one place the vuln-only posture is relaxed for cargo-deny specifically, mirroring osv-scanner's `ignored-advisories.toml` for its own (non-vulnerability) exceptions.

### Cross-ecosystem dependency SCA — osv-scanner + trivy-fs (source scope, always run)

The non-Rust counterpart to cargo-deny, extended to every ecosystem and run unconditionally over the source tree's committed lockfiles. Both fetch **fresh** advisory/license data at scan time and fail closed if they can't (see [ADR-030](decisions/030-cross-ecosystem-sca.md)).

- **osv-scanner** (`osv-scanner scan source -r`) — the cargo-deny analog in one pass. **Vulnerabilities:** any advisory in OSV.dev fails — **all severities, including unfixable** (OSV-Scanner has no severity floor, which is exactly the cargo-deny "any hit fails" model). Covers npm, PyPI, Go, Maven, RubyGems, Cargo, and more from their lockfiles. **Licenses:** not enforced ([ADR-034](decisions/034-sca-vuln-only-license-loosening.md) — `--licenses` dropped). **Unmaintained advisories:** osv-scanner has no advisory-type filter, so the gate appends a fleet-owned data file (`config/util/osv-scanner/ignored-advisories.toml`, a short justified `[[IgnoredVulns]]` list) to its generated config — the one place the vuln-only posture is relaxed. `--allow-no-lockfiles` lets a depless repo (pure Ansible/YAML) pass.
- **trivy-fs** (`trivy fs`) — a second, broader vuln source run alongside osv-scanner for **union coverage**. Trivy's DB aggregates GitHub Security Advisories + OSV + vendor feeds, so it catches advisories OSV.dev alone may not. All severities, `library` packages, `ignore-unfixed: false`. Source-scope, so it catches vulnerable deps **pre-build**, independent of whether the dependency survives into the final image (which the image-scope Trivy scanner checks separately).

Together these close the old "no SCA outside Rust" gap: a vulnerable or disallowed-license dependency in any ecosystem fails the gate at the source, before the image is ever signed. The most aggressive — and most likely to need per-ecosystem tuning — is the license allow-list (deny-by-default + deny-unknown) and the deny-unfixable-vuln stance.

### Rust correctness/safety — clippy + rustfmt + test (`Cargo.toml` marker)

`cargo clippy --all-targets --all-features` with `-D warnings` (warnings are blockers) and `clippy::pedantic` on. The denied lints map directly to failure classes:

- **Runtime panics in production paths:** `unwrap_used`, `expect_used`, `panic`, `get_unwrap`, `indexing_slicing`, `string_slice` denied — code that can panic on bad input is rejected (tests are exempted via `clippy.toml`, where panics are legitimate).
- **Memory-safety escape hatch:** `-D unsafe_code` — no `unsafe` blocks reach production without a scoped `#[expect]`.
- **Silent error swallowing:** `map_err_ignore`, `let_underscore_must_use` denied — errors can't be discarded unseen.
- **Unfinished/debug code shipping:** `todo`, `unimplemented`, `dbg_macro` denied — placeholder code and debug prints don't pass.
- **Suppression hygiene:** `allow_attributes` and `allow_attributes_without_reason` denied — you cannot blanket-`#[allow]` your way past a lint; you must use `#[expect(..., reason = "…")]`. This is what makes the whole regime tamper-evident.
- **Assorted footguns:** `clone_on_ref_ptr`, `rc_buffer`, `rc_mutex` (wrong-smart-pointer bugs), `wildcard_enum_match_arm` (new enum variants silently unhandled), `shadow_unrelated` (accidental variable shadowing), `format_push_string`, `verbose_file_reads`, and more.

`cargo fmt --check` enforces consistent formatting (a quality/readability floor, not a security control). `cargo test --all-targets` runs the repo's tests — the gate fails if the suite fails, but it does not enforce that meaningful tests *exist* (see gaps).

### Python — deps + ruff + mypy + pytest (`pyproject.toml` marker)

- **deps** installs the repo's dependencies into a per-project `.venv` (via `uv`) *before* mypy/pytest run — the Node `npm ci` analog. Without it, third-party imports resolve to `Any` and mypy `strict` cascades (e.g. `Class cannot subclass "BaseModel"` for every Pydantic model), and the repo's own package isn't importable so pytest collection fails. `uv` is a gate-side installer using its pip interface against the repo's standard `requirements.txt`/`pyproject.toml` — a scanned repo never has to adopt uv. First-party resolution is hybrid: an installable repo (`[project]`/`[build-system]`) is editable-installed (covers `src/` layouts); a loose tree resolves by rootdir via `pythonpath = .`. A deps-install failure fails the step closed. See [ADR-032](decisions/032-python-dependency-install.md). (`.venv` is in `skip_dirs` so it's honest-map-safe, and it's excluded from trivy-fs so the gate-injected pytest can't be mis-attributed as a repo dependency CVE.)
- **ruff `select = ["ALL"]`** with a minimal conflict-only ignore list — maximum lint enforcement, every rule group on. This includes ruff's security group (`S`, the bandit port): catches `assert` in production, hardcoded passwords, `subprocess` with `shell=True`, `eval`/`exec`, insecure hashlib usage, unsafe YAML/pickle loads, bind-all-interfaces, insecure temp files, and SSL-verification-disabled patterns. Also the bug-risk groups (`B`), comprehension/perf, and full docstring/annotation discipline. Tests are scoped-exempted for asserts, magic values, and missing annotations/docstrings so the regime stays usable. Because ruff has no warn/error split, any selected violation fails — "warnings are blockers" is automatic.
- **mypy `strict = True`** plus `warn_unreachable`, `warn_redundant_casts`, `warn_unused_ignores`, `disallow_any_generics`, `extra_checks` — prevents whole classes of type-confusion bugs, unreachable/dead code, and stale `# type: ignore` suppressions. `--python-executable` points mypy at the repo's installed `.venv` so its real (`py.typed`) third-party types resolve. `ignore_missing_imports` stays on so the gate doesn't fail purely because a third-party lib ships no stubs (not a code-quality signal).
- **pytest** runs the suite from the repo's `.venv` (`--strict-markers` so unregistered markers error). Same caveat as Rust: it runs tests, it doesn't mandate good ones. Gate `pythonpath = .` resolves first-party imports for the loose-layout case. Test *discovery* (`testpaths`) is the acknowledged repo-shaped fragile case (noted in `pytest.ini`).

### Node/TS (+ React + Svelte) — eslint (type-aware) + tsc + svelte-check (`package.json` marker)

`npm ci` installs deps (needed for type resolution), then eslint and tsc run from the gate's own pinned toolchain (`eslint` + `typescript` + `typescript-eslint`, installed into the gate's node config dir so the flat config can resolve the plugin). Three layers, reaching Rust/Python-grade strictness for TypeScript:

- **eslint — security/discipline core (all JS + TS):** blocks the eval-class footguns (`no-eval`, `no-new-func`, `no-script-url` — the JS analog of `unsafe_code`), debug leftovers (`no-debugger`, `no-alert`), and discipline rules (`eqeqeq`, `no-var`, `prefer-const`, `no-param-reassign`, `no-template-curly-in-string`, `no-implicit-globals`). These have no typescript-eslint equivalent, so they apply to every file.
- **eslint — JS correctness (JS only):** likely-bug rules (`no-undef`, `no-unused-vars`, `no-fallthrough`, `no-self-compare`, `use-isnan`, `valid-typeof`, `no-cond-assign`, `no-implied-eval`, `no-empty` with `allowEmptyCatch: false`, `no-unused-expressions`, `no-throw-literal`, `require-await`, `no-shadow`). TypeScript files get the type-aware equivalents instead (below), so applying the base rules to TS would double-report.
- **eslint — type-aware strictness (TS only):** `typescript-eslint` `strictTypeChecked` + `stylisticTypeChecked` — the JS analog of clippy pedantic: `no-floating-promises`, `no-misused-promises`, the `no-unsafe-*` family, `require-await`/`only-throw-error` with type info, etc. These need a TypeScript program, supplied by the **same wrapped tsconfig tsc uses** (gate owns the rules; the repo's tsconfig provides type/module resolution). A node project containing TypeScript **must declare its `tsconfig`** in `gate-config.json` — the harness fails closed otherwise — so the type program is always available.
- **tsc `--noEmit` + full `strict` family** — strict null checks and complete type strictness. By default runs against the gate's `tsconfig.base.json`; when a node project sets `tsconfig`, the harness wraps the repo's own: a generated config `extends` it (inheriting the repo's `paths`/`baseUrl`/module resolution) while force-overriding the entire `strict` family locally — so the typecheck is *project-aware* **and** the repo cannot opt out of any sub-flag (`extends` is overridden per-key; tsc's default `include` means the repo can't shrink the typechecked set). See [ADR-029](decisions/029-gate-config-honest-map.md).
- **React (`.jsx`/`.tsx`)** — needs no separate adapter (React is node), and gets a React-specific eslint layer: `react-hooks` `rules-of-hooks` (a hook called conditionally / in a loop is a real runtime bug) and `exhaustive-deps` (stale-closure effects) are **errors**, plus `eslint-plugin-react`'s recommended correctness set. `prop-types` is off (tsc / type-aware eslint already enforce prop typing) and the modern automatic JSX runtime means `react-in-jsx-scope` is off. So JSX correctness is held to the same blocking bar as the rest of the node tier.
- **Svelte (`.svelte`)** — `tsc` cannot parse single-file components, so `svelte-check` is the component typechecker, run against the **same wrapped tsconfig** tsc uses (the gate's forced strictness applies to component `<script lang="ts">`; the repo's `svelte.config.js` supplies preprocessing only). Because a component's TypeScript is invisible to the `.ts`/`.tsx` extension rule, the honest map content-sniffs `.svelte` files: a component with `<script lang="ts">` makes its node project **require a declared `tsconfig`** or the gate fails closed — the exact analog of the TS-without-tsconfig rule. `eslint-plugin-svelte`'s recommended ruleset lints component markup/logic, with the script parser set to typescript-eslint so `lang="ts"` parses. See [ADR-031](decisions/031-go-svelte-react-coverage.md).

⚠️ **Remaining nuance:** The one source-policy check it still lacks versus Rust is dependency *source/registry* gating (cargo-deny's `unknown-registry`/`unknown-git`/`wildcards`) — see the gaps list.

### Go — golangci-lint + gofumpt + govulncheck + go test (`go.mod` marker)

The gate's first compiled non-Rust language, mirroring the Rust adapter's shape and reaching for the same max-strictness floor. `golangci-lint run` is forced onto the gate's own `.golangci.yml` via `-c` (which also disables discovery of the repo's config — the config-forcing guarantee), with `default: none` and an explicit aggressive enable list so an upstream default change can't silently weaken it. golangci-lint exits non-zero on any finding, so "warnings are blockers" is automatic. The enabled linters map to the same failure classes as the Rust section:

- **Silent error swallowing / error mishandling:** `errcheck` (unchecked error returns), `errorlint` (wrong error wrapping/comparison), `nilerr` (returning `nil` after a non-nil error) — the Go analog of `map_err_ignore`/`let_underscore_must_use`.
- **In-language security:** `gosec` — the bandit/`ruff S` analog. Catches hardcoded credentials, weak crypto (MD5/SHA1/DES), `math/rand` for secrets, command injection, path traversal, insecure file perms, disabled TLS verification, and (`G103`) `unsafe` usage — the closest Go has to Rust's `-D unsafe_code`.
- **Correctness + pedantic:** `govet`, `staticcheck` (v2 folds in gosimple + stylecheck), `revive`, `gocritic` — the `clippy::pedantic` analog.
- **Footguns & resource leaks:** `ineffassign`, `unused`, `bodyclose` (unclosed HTTP bodies), `noctx` (context-less requests), `rowserrcheck`/`sqlclosecheck` (DB hygiene), `makezero`, `predeclared`.
- **Suppression hygiene:** `nolintlint` with `require-explanation` + `require-specific` — a `//nolint` must name a specific linter **and** carry an explanation, so you cannot blanket-`//nolint` past the gate. This is the `allow_attributes_without_reason` analog that keeps the regime tamper-evident.
- **Tests** are scope-exempted from `gosec`/`errcheck` (fixture credentials, deliberately-ignored errors are legitimate in tests), mirroring the Rust `clippy.toml` test exemptions.

`golangci-lint fmt --diff` enforces **gofumpt** + **goimports** formatting (gofumpt is the stricter superset of gofmt; the `rustfmt --check` analog). `govulncheck ./...` is a call-graph-aware dependency vuln scan: it fetches a fresh vuln DB at scan time (the deliberate SCA-freshness stance — a newly-disclosed advisory fails a previously-green build) and only flags advisories actually reachable from the code, so it complements the always-on osv-scanner + trivy-fs (which also cover `go.mod`/`go.sum` from the lockfile). `go test ./...` compiles every package and runs the suite — same caveat as Rust/Python: it runs tests, it does not mandate good ones. See [ADR-031](decisions/031-go-svelte-react-coverage.md).

### Ansible — ansible-lint (`ansible/` dir marker)

`ansible-lint --strict` on the **`production` profile** — ansible-lint's strictest built-in bar, nothing higher exists. `--strict` makes warnings fail. Prevents the common Ansible failure/security classes the production profile encodes: missing `no_log` on tasks handling secrets, use of `command`/`shell` where a module exists, unpinned/`latest` package installs, world-readable file modes, missing `become` discipline, deprecated syntax, and idempotency violations. This directly backstops the platform's "secrets must never appear in plaintext in IaC" rule.

### YAML — yamllint (`.yamllint` marker, opt-in)

`yamllint --strict` on a tightened default ruleset. A repo opts in by shipping a `.yamllint` marker, but the *rules* applied are the gate's, not the repo's. Catches structural YAML hazards: duplicate keys, the **implicit/explicit octal** trap (`forbid-implicit-octal`/`forbid-explicit-octal` — the classic file-mode `0644`-vs-`644` footgun), non-canonical truthy values, document-start consistency, and line-length (error at 160). This is a correctness floor for the platform's enormous surface of Helm/Flux/CI YAML.

## What the gate does NOT prevent (gaps & non-goals)

Being explicit here matters more than the coverage list — these are the things a green gate does *not* promise, and treating them as covered is how something slips through.

- **It does not verify the gate image or signing key.** A compromised `grizzly-gate` image or a leaked cosign private key defeats everything. That trust root is protected by pinning + secret-store access control + key rotation (see the [platform runbook](https://github.com/Grizzly-Endeavors/grizzly-platform/blob/master/docs/runbooks/ci-gate.md)), not by any check here.
- **It does not mandate meaningful tests.** `cargo test`/`pytest` fail if the existing suite fails, but a repo with zero or trivial tests passes the test step. The gate enforces "tests don't regress," not "behavior is tested." Coverage thresholds are not enforced.
- **Semgrep ruleset is pinned to a branch, not a SHA.** `SEMGREP_RULES_REF=develop` means two builds of the same gate Dockerfile at different times can vendor different rules. Reproducibility holds *within* a built gate tag, not across rebuilds of the same tag. (TODO in the Dockerfile.)
- **The image scanner's `ignore-unfixed: true` lets unpatchable HIGH/CRITICAL CVEs ship.** Deliberate, and scoped to the *image* scanner (base-OS/bundled CVEs with no fix shouldn't block deploys indefinitely) — so the built image is only guaranteed free of *fixable* HIGH/CRITICAL vulns. Note the source-dep SCA scanners make the opposite trade (`trivy-fs` is `ignore-unfixed: false`, osv-scanner has no severity floor), so an unfixable vuln in a *declared dependency* does fail.
- **SCA depends on committed lockfiles and third-party metadata.** Cross-ecosystem SCA now exists (osv-scanner + trivy-fs — see that section), closing the old "no SCA outside Rust" gap. Residual limits: it reads **committed lockfiles**, so a repo that doesn't commit a lockfile gets no dependency resolution to scan; advisory coverage is only as complete as OSV.dev + Trivy's DBs; and license verdicts depend on deps.dev's license metadata (a package deps.dev can't classify is denied as `non-standard`, which is intentional but can false-positive). The `unknown-registry`/`unknown-git`/`wildcard` *source* checks remain Rust-only (cargo-deny) — there is no equivalent "dependency came from an untrusted source" gate for npm/PyPI/Go.
- **No IaC/Kubernetes-manifest security scanning.** Trivy runs in `image` mode only here; there is no `trivy config` / Checkov / kube-linter step, so misconfigured K8s manifests, Dockerfiles, or Terraform are not scanned for security posture by the gate (Kyverno enforces a separate set of admission policies at deploy time).
- **No runtime, behavioral, or business-logic review.** The gate is static + scan-based. Logic errors that are syntactically clean, type-correct, and pattern-free pass. "The gate is the reviewer" means the reviewer is a very strict linter, not a human who understands intent.
- **Detection is extension/shebang-based, bounded by the denylist.** Honest-map verification closes the old presence-based scope hole, but its completeness rests on the detection ruleset. A *code* language that is neither adapter-backed nor on the `detect.toml` denylist (seeded with ruby/java/kotlin/scala/c/cpp/csharp/php/swift/perl/lua/elixir/dart) would not be flagged — extend the denylist as new languages appear. Likewise, code hidden under an Ops-owned `skip_dirs` name (`node_modules/`, `target/`, …) is not adapter-checked (though still secret/SAST-scanned); that list is the one detection boundary an attacker could lean on, so it's kept tight. And `ansible`/`yaml` remain opt-in markers, not extension-detected — a repo with Ansible content but no declared `ansible` project gets no ansible-lint (a bare `.yml` is too ambiguous to mandate).

## Keeping this doc honest

This file enumerates behavior derived from a specific config snapshot. When you change a lint level, add/remove a tool, bump a pin, or alter a scanner's severity/scope, update the matching section here in the same change — and bump the gate tag. If a claim in this doc and the config in `config/` disagree, the config wins and this doc is the bug.
