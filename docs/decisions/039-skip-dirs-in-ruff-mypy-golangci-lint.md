# ADR-039: Wire skip_dirs into ruff, mypy, golangci-lint, tsc, and svelte-check

**Date:** 2026-07-07
**Status:** accepted

## Context

The eslint skip_dirs fix (`config/languages/node/manifest.toml`, `eslint.config.mjs.tmpl`) single-sources the Ops-owned `skip_dirs` (`detect.toml`) into eslint via the harness's general `{skip_dirs}` substitution token → `GATE_SKIP_DIRS` env → a global ignores block. That closed a real gap: ESLint 9 flat config only auto-ignores `node_modules`, so it was linting `build`/`dist`/`.svelte-kit`/vendor — dirs the honest-map walk does not treat as first-party code. The other tree-walking adapters (ruff, mypy, golangci-lint) had no equivalent wiring and were relying on whatever each tool's own defaults happen to exclude — audited here.

**ruff** (0.15, verified via `ruff check --show-settings`): its own default `exclude` list already covers `.venv`/`venv`/`.mypy_cache`/`.pytest_cache`/`.ruff_cache`/`.tox`/`dist`/`node_modules`, but **not** `target`/`vendor`/`__pycache__`/`.cargo`/`.gradle`/`build`/`.svelte-kit` — 7 of the gate's 15 skip_dirs entries. `build`/`dist` matter concretely: they're the standard `setuptools`/`python -m build` output dirs, and `dist` is covered by ruff's default but `build` is not.

**mypy** (2.1.0): has no default excludes at all — everything under the tree it's pointed at is checked unless `--exclude` is set.

**golangci-lint** (pinned `2.12.2`): this is where the audit found real, non-obvious, version-specific behavior worth documenting rather than guessing at:

1. **The v1 mechanism is gone.** `run.skip-dirs` and `issues.exclude-dirs`/`issues.exclude-dirs-use-default` were removed in the v2 config schema (confirmed against the v2 migration guide and `.golangci.reference.yml`); there is no `--skip-dirs` CLI flag in `golangci-lint run --help` for 2.12.2. The v2 replacement is `linters.exclusions.paths` (for `run`) and the separate `formatters.exclusions.paths` (for `fmt`) — two independent config-file-only lists.
2. **No env-var expansion.** golangci-lint's config file only supports the `${base-path}`/`${config-path}` placeholders, not arbitrary env vars — so there is no way to hand it the harness's `{skip_dirs}` token at run time the way eslint/ruff/mypy receive it. The list has to be static and hand-maintained.
3. **`vendor` is not auto-excluded the way "everyone knows" it is.** `go help packages` documents that `./...` doesn't match packages under a directory literally named `vendor`, but this only applies to the module's canonical top-level vendor directory. Verified empirically (spun up a real `go.mod` + golangci-lint 2.12.2 binary): a directory named `vendor` anywhere in the tree, and `node_modules`/`target`/`build` too, all show up in `go list ./...` and get linted/format-checked like any other package. Only dot-/underscore-prefixed dirs and `testdata` are skipped by Go tooling itself.

**tsc / svelte-check** (`typescript` 6.0.3, `svelte-check` 4.7.1, both pinned in the `Dockerfile`): unlike the other three, these are not CLI-flag/env-var driven — they're `include`/`exclude` driven through the (possibly repo-owned) tsconfig the harness already wraps in `resolve_tsconfig` (`harness/src/main.rs`) via `"extends"`. This raised a distinct risk worth verifying empirically before touching it: **does TypeScript's `extends` merge `include`/`exclude` between parent and child, or does the child's value replace the parent's?** Confirmed against the pinned `6.0.3` binary with `tsc --showConfig --project <config>` (prints the fully resolved effective config without emitting anything):

- A "repo" tsconfig with `"exclude": ["something-else"]`, wrapped by a child that itself sets `"exclude": ["dist"]` — resolved `exclude` is `["dist"]` only. **The child's value fully replaces the parent's; they are not merged.** Naively adding `"exclude": [...skip_dirs]` to the wrapper would therefore have silently dropped `something-else`, a real correctness regression, not a fix.
- The same repo tsconfig wrapped by a child that sets **no** `exclude` at all — resolved `exclude` is `["something-else"]` (the parent's, inherited as-is), and a real `tsc --noEmit` run confirms a planted type error under a would-be skip_dir (`dist/generated.ts`) **is** typechecked and fails the build — this is the bug the issue reports, reproduced.
- `svelte-check --tsconfig` documents "only files matched by the files/include/exclude pattern... are diagnosed" and was verified empirically to delegate to the exact same tsconfig resolution as `tsc` — a `.svelte` component under a would-be skip_dir is typechecked/fails when the wrapper has no `exclude`, and is correctly skipped once the wrapper's `exclude` covers that dir (same wrapper file, so the fix for `tsc` and `svelte-check` is one and the same).
- `tsc --showConfig` also tolerates JSONC (`//` and `/* */` comments, trailing commas — verified with a repo tsconfig containing both) and resolves a **transitive** `extends` chain (a repo tsconfig that itself `extends` a grandparent owning the `exclude`, with no `exclude` of its own — verified the grandparent's `exclude` still resolves correctly). Both matter because a real-world repo's tsconfig.json commonly has comments and sometimes extends a shared base.

## Decision

1. **ruff**: added `--extend-exclude {skip_dirs}` to the check's `cmd` (`config/languages/python/manifest.toml`). Deliberately `--extend-exclude`, not `--exclude` — verified that `--exclude` **replaces** ruff's own default exclude list rather than adding to it (`ruff check --exclude ... --show-settings` shows `file_resolver.exclude` reduced to only the passed value), which would have silently dropped `.git`/`.venv`/etc. `--extend-exclude` correctly layers on top.
2. **mypy**: added `--exclude` built from `GATE_SKIP_DIRS` (env, same mechanism as eslint) via a `sh -c` wrapper that turns the comma list into a `(^|/)(a|b|c)(/|$)` regex alternation with `tr`, since mypy's `--exclude` takes a single regex (no comma-list form like ruff). Verified mypy's `--exclude` only affects recursive directory discovery — irrelevant here since the check always hands mypy `.` and lets it discover the tree itself, never explicit file args.
3. **golangci-lint**: no run-time substitution is possible (see Context), so `config/languages/go/.golangci.yml` gets a **static, hand-maintained mirror** of `detect.toml`'s `skip_dirs`, listed explicitly under both `linters.exclusions.paths` and `formatters.exclusions.paths` (verified both independently with a real Go module + the pinned 2.12.2 binary — `run` and `fmt --diff` each correctly stop reporting on the listed dirs while still failing on first-party violations). Every skip_dirs name is listed explicitly rather than relying on partial Go-tooling defaults, given the `vendor` surprise above.
4. **tsc / svelte-check**: `resolve_tsconfig` (`harness/src/main.rs`) now sets the wrapper's `"exclude"` to the Ops-owned `skip_dirs` **union** the repo's own tsconfig's resolved `exclude` — never skip_dirs alone, per the replace-not-merge finding above. The repo's own resolved excludes are obtained by shelling out to the same pinned `tsc --showConfig --project <repo tsconfig>` used for empirical verification, rather than hand-parsing the repo's tsconfig.json with `serde_json`: this reuses the exact tool (and thus exact JSONC + transitive-`extends` handling) that will typecheck the project, and avoids adding a JSONC-parsing dependency to the harness for a one-off read. When no repo tsconfig is declared, the existing "use `tsconfig.base.json` as-is" branch is untouched — that project (by the honest-map's own TS-without-tsconfig rule) has no TypeScript to typecheck, so there is nothing for an exclude to protect. No `manifest.toml` change was needed for `tsc`/`svelte-check` (unlike ruff/mypy/eslint): their injection point is the wrapped tsconfig file itself, already threaded through both `cmd`s via the existing `{tsconfig}` token, so the fix lives entirely in `resolve_tsconfig`.

## Consequences

- ruff/mypy/golangci-lint/tsc/svelte-check now skip the same generated/vendor dirs the honest-map walk does, closing the class of false positives (and false confidence) the eslint fix already closed for node.
- The golangci-lint list is **not** single-sourced — it's a manually-maintained duplicate of `detect.toml`'s `skip_dirs`, kept in sync by hand (mirrors the precedent in [ADR-038](038-exclude-generated-output-from-source-scanners.md), where the semgrep/gitleaks generated-output exclude list is likewise separate and hand-maintained, for a different reason). A future `skip_dirs` addition in `detect.toml` does **not** automatically propagate to `.golangci.yml`; a test (`golangci_lint_config_excludes_skip_dirs` in `harness/src/config.rs`) catches drift between the two lists at the *name* level, but not a genuinely new golangci-lint CLI/config mechanism arriving later that would let this be single-sourced properly.
- `resolve_tsconfig` now spawns an extra `tsc --showConfig` per node project (before the real `tsc`/`svelte-check` checks run) purely to resolve the repo's own `exclude`. This is a second process spawn, not a new dependency — the same pinned `tsc` binary the check already uses — and it fails the run closed (an error, not a silently-empty exclude) if the repo's own tsconfig can't be resolved this way, since the real `tsc`/`svelte-check` checks would fail identically against the same broken config moments later.
- Known limitation, not fully closed: this only resolves `include`/`exclude` through tsconfig's own `extends` chain (which `tsc --showConfig` does transitively and correctly — verified). It does **not** attempt anything beyond what `tsc --showConfig` itself resolves — if a future TypeScript version changes how `--showConfig` reports `exclude`, this wiring rides on that contract rather than re-implementing it.
- gitleaks/semgrep/trivy-fs/osv-scanner remain unwired on purpose — they intentionally walk the whole tree (including all of `skip_dirs`) for secrets/SAST/SCA, per `detect.toml`'s existing comment and ADR-038's narrower exclude list. This ADR does not touch that.
